## 8. Lessons Learned

This is the first Development Sprint in the DT-017–DT-026 sequence to add zero production code, and it validates a pattern this engagement has applied since DT-011/DT-015 (composition/verification-only Development Tasks): a plan that identifies, before implementation, exactly which existing mechanism will satisfy the Acceptance Criteria (here, `AssignmentValidator`'s already-existing `caller.organizationId !== assignment.organizationId` comparison) allows the "no production code change required" expectation to be stated with evidence rather than hope — and this closure confirms that expectation held exactly as predicted, not merely "close enough."

A second finding concerns the value of the "escalate rather than silently fix" instruction (`Development_Sprint_019_Plan.md` Section 18, a novel addition to this engagement's Development Agent Prompt template): because no production code change proved necessary, this instruction's actual effect cannot be observed directly in this sprint's outcome — but its presence in the plan is itself evidence of a maturing discipline: rather than assuming a verification-only task will never surface a real gap, the plan explicitly prepared for that possibility and defined the correct response (stop and report) in advance, consistent with this repository's broader "Escalate Instead of Guessing" principle (EP-008 Ch01 §5.x). This is recorded here as a template improvement worth carrying into future verification-only tasks, independent of whether this specific sprint needed it.

A third, smaller finding: the new-test-file judgment call disclosed at planning time (`Development_Sprint_019_Plan.md` Section 11 — creating `OrganizationOwnedScanPipeline.test.ts` rather than extending `NfcScanToTimeEntryPipeline.test.ts`) was implemented exactly as the plan recommended, and this closure independently confirms the existing file was not touched (Section 2). This is a second, concrete instance of a disclosed, reversible planning judgment call being carried through to implementation unchanged, once no Technical Lead objection was raised — the same pattern this engagement has now observed multiple times (e.g. DT-025's missing-target-Customer resolution, `Development_Sprint_018_Closure.md` Section 8).
