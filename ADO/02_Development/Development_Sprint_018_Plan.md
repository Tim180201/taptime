## 1. Executive Summary

Development Sprint 017 completed DT-024, extending `OrganizationAdministrationService` with its second method, `registerNfcTag(...)`. Development Sprint 018 continues the TS-002 implementation sequence with the next unblocked unit: **DT-025 — Organization Administration: NFC Tag Assignment — and DT-025 only**. No other Development Task (DT-026) is part of this sprint.

DT-025 extends the existing `OrganizationAdministrationService` — created by DT-023, extended by DT-024, not created again here — with a third method, `assignNfcTag(...)`, completing the three-method service TS-002 planned for Capability 4 (Administration). `assignNfcTag(...)` is materially more complex than `createCustomer(...)`/`registerNfcTag(...)`: beyond the shared `MembershipAuthorizationValidator` orchestration, it must additionally verify that a caller-supplied `NfcTag` and `AssignmentTarget` both belong to the same Organization as the requesting Membership before constructing an `NfcAssignment`. This plan's primary contribution is resolving exactly how that additional cross-check is implemented without changing `AssignmentTarget`'s shape, without adding any repository method, and without inventing a new result category — see Section 2 (Repository Evidence) and Section 3 for the reasoning.

After this sprint, an Administrator Membership can, for the first time, assign an already-registered NFC Tag to an already-created Customer through a real Application Service call — completing, at the orchestration level only, the create → register → assign chain TS-002 describes — though still only from a test, since no UI, CLI, or mobile entry point calls `OrganizationAdministrationService` yet, and no scan-pipeline integration verification (DT-026) has been performed against this data.

## 5. Business Objective

Give an Administrator Membership a real, callable way to assign an already-registered NFC Tag to an already-created Customer within their own Organization — the third and final Application Service method TS-002 planned for Organization Administration, and the first to combine authorization with a same-Organization cross-check spanning two other domain objects (`NfcTag`, `AssignmentTarget`/`Customer`) in addition to the requesting Membership itself. This completes, at the orchestration level, the create → register → assign chain FB-002/TS-002 describe. It is a genuine step toward Organization Management usability (Section 14, below), but it does not by itself make TapTim.e pilot-ready: no UI, CLI, or mobile entry point calls `OrganizationAdministrationService` after this sprint, so an NFC Tag can still be assigned only from a test, and DT-026's own scan-pipeline verification against Administration-flow-written data has not yet been performed.

## 6. Technical Objective

Extend `OrganizationAdministrationService` with exactly one new method, `assignNfcTag(...)`, that: (1) calls `MembershipAuthorizationValidator.authorize(membership, organizationId)` first; (2) on rejection, returns the rejection reason and performs no write; (3) on acceptance, verifies the caller-supplied `NfcTag`'s `organizationId` matches the accepted Membership's own `organizationId`, rejecting with `cross_organization_access` and performing no write if not; (4) resolves the caller-supplied `AssignmentTarget`'s owning `Customer` via `CustomerRepository.findById(target.targetId)` and verifies that `Customer`'s `organizationId` matches the accepted Membership's own `organizationId` (treating a missing `Customer` as an organization mismatch — Section 8), rejecting with `cross_organization_access` and performing no write if either check fails; (5) on full acceptance, constructs an `NfcAssignment` with `organizationId` taken from the requesting Membership's own `organizationId`, `nfcTagId` taken from the supplied `NfcTag.id`, `target` taken from the supplied `AssignmentTarget` unchanged, and `active: true`, calls `NfcAssignmentRepository.save`, and produces `NfcTagAssigned`. Zero change to `MembershipAuthorizationValidator`, `MembershipAuthorizationResult`, `Membership`, `NfcTag`, `AssignmentTarget`, `Customer`, `NfcAssignment`, `CustomerRepository`, `NfcTagRepository`, `NfcAssignmentRepository`, `InMemoryCustomerRepository`, `InMemoryNfcTagRepository`, `InMemoryNfcAssignmentRepository`, `NfcTagAssigned`, `createCustomer(...)`'s own existing behavior, or `registerNfcTag(...)`'s own existing behavior — every one of these is consumed exactly as it exists today.

**AssignmentTarget/NfcTag resolution scope note:** per the assigning task's own guidance, `assignNfcTag(...)` does not call `NfcTagRepository` at all — the `NfcTag` argument is taken as an already-resolved value supplied by the caller, exactly mirroring how `registerNfcTag(...)` accepts an already-normalized `NfcPayload` rather than re-deriving one (Section 2/3). `AssignmentTarget`'s shape is not modified; only its existing `'customer'` variant is handled, resolved via the already-injected `CustomerRepository.findById(...)`, mirroring `AssignmentValidator`'s own precedent exactly (Section 2). No non-customer `AssignmentTarget` behavior is invented.

**Planning Finding — missing-target rejection reason (required disclosure per the assigning task's own guidance):** repository evidence does not define a distinct rejection reason for a target `Customer` that cannot be found by `CustomerRepository.findById(...)` within this administrative flow. `MembershipAuthorizationResult`'s three existing rejection reasons (`membership_not_found`, `membership_lacks_administrator_role`, `cross_organization_access`) contain no "target not found" category, and this task's Architecture Boundaries do not permit modifying `MembershipAuthorizationResult` to add one (it is a consumed-only component). A structurally similar reason, `missing_assignment_target`, exists on `AssignmentValidationResult` (`AssignmentValidator`'s own result type, used at FB-001 scan time, Section 2) — but importing or reusing that reason here would require either modifying `MembershipAuthorizationResult` (forbidden) or introducing a second, wrapper-style result type carrying a broader reason vocabulary (a design the assigning task's guidance discourages: "prefer existing, repository-approved result semantics... do not invent unrelated result categories unless repository evidence requires and the gap is documented"). FB-002 Decision 5's own product-level Result line — "`NfcTagAssigned`, or `CrossOrganizationAccessRejected`" — names only two outcomes, with no third category, at the same precision level Decision 3/4 used for `createCustomer(...)`/`registerNfcTag(...)`, both already resolved by reusing `MembershipAuthorizationResult`'s existing shape rather than inventing a new one. **Resolution adopted by this plan**: a target `Customer` that cannot be found is treated as failing the same-Organization check and rejected with `cross_organization_access`, the same reason used for an actual cross-Organization mismatch — a `Customer` that does not exist cannot be verified as belonging to the requesting Membership's Organization, so treating "not found" and "found but wrong Organization" identically is a defensible, narrow reading of "belongs to the same Organization" being false in both cases, and it requires no new type or reason. This is a disclosed interpretive decision, not a silent one: the Development Agent Prompt (Section 17) states it explicitly, a dedicated test is required to prove it (Section 12), and this closure-time finding should be re-confirmed by the Technical Lead/Human Architect as correct rather than assumed self-evident.

export type AssignNfcTagResult =
  | { readonly status: 'accepted'; readonly nfcAssignment: NfcAssignment; readonly event: NfcTagAssigned }
